System Architecture

How the pieces fit together — review the flow, then we'll talk about it in the terminal

Components & data flow
1 · WhatsApp Worker (always-on)

Baileys client, hosted free on Render/Railway/Fly.
Linked to each community's WhatsApp via QR.
Listens to the group → on each new message, forwards raw text + media to the web app's secure webhook.

→
2 · Next.js App on Vercel

• /api/ingest webhook → Claude Haiku parses Hebrew text into fields → saves a draft candidate (pending review), uploads photo to Blob.
• Web UI (Hebrew, RTL): review drafts, search, candidate profiles, match tracking, deactivation, settings.
• Auth.js: magic-link + Google.

→
3 · Data stores (free tier)

• Vercel/Neon Postgres — candidates, matches, communities, users (Prisma ORM).
• Vercel Blob — candidate photos.

Matchmaker opens the web app → reviews/edits the parsed draft → approves it into the candidate pool → searches for matches → records suggestions & their status → deactivates candidates who are no longer available.
Multi-tenant: every row is scoped to a community. A matchmaker only ever sees their own community's data.
